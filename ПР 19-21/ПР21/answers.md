1. Какие основные принципы REST архитектуры вы знаете?
Основные принципы REST архитектуры:
Единообразие интерфейса (Uniform Interface) - стандартизированный способ взаимодействия между клиентом и сервером
Отсутствие состояния (Stateless) - каждый запрос содержит всю необходимую информацию для его обработки
Кэшируемость (Cacheable) - ответы могут быть закэшированы для улучшения производительности
Клиент-серверная архитектура - разделение ответственности между клиентом и сервером
Слоистая система (Layered System) - архитектура может состоять из нескольких уровней
Код по требованию (Code on Demand, опционально) - сервер может предоставлять исполняемый код клиенту

2. В чем разница между HTTP методами PUT и PATCH?
PUT - используется для полного обновления ресурса. Клиент должен отправить полное представление ресурса
PATCH - используется для частичного обновления ресурса. Клиент отправляет только изменяемые поля

3. Какие коды состояния HTTP наиболее часто используются в REST API?
Наиболее часто используемые HTTP коды состояния:
200 OK - успешный запрос
201 Created - ресурс успешно создан
400 Bad Request - некорректный запрос
401 Unauthorized - требуется аутентификация
403 Forbidden - доступ запрещен
404 Not Found - ресурс не найден
500 Internal Server Error - внутренняя ошибка сервера

4. Что такое ресурсо-ориентированный дизайн URL?
Ресурсо-ориентированный дизайн URL предполагает использование существительных для обозначения ресурсов в URL, а глаголы (действия) выражаются через HTTP методы. Например:
GET /api/books - получить список книг
POST /api/books - создать новую книгу
GET /api/books/1 - получить книгу с ID 1
PUT /api/books/1 - обновить книгу с ID 1

5. Какие преимущества предоставляет пагинация в API?
Преимущества пагинации:
Улучшение производительности - уменьшение объема передаваемых данных
Снижение нагрузки на сервер - обработка меньших наборов данных
Улучшение пользовательского опыта - постепенная загрузка контента
Оптимизация использования памяти - как на клиенте, так и на сервере
Удобств навигации - четкая структура данных по страницам

6. Как организовать валидацию данных в REST API?
Способы организации валидации:
Встроенная валидация Express через middleware
Кастомные функции валидации для проверки обязательных полей и форматов
Сторонние библиотеки как Joi, Yup, express-validator
Валидация на уровне схемы данных при использовании баз данных
Проверка уникальности полей в базе данных

7. Какие подходы к обработке ошибок в REST API вы знаете?
Подходы к обработке ошибок:
Централизованный обработчик ошибок через middleware
Стандартизированный формат ответов об ошибках
Использование соответствующих HTTP статус-кодов
Подробное логирование ошибок для отладки
Гранулярные сообщения об ошибках для разных типов сбоев
Обработка необработанных исключений для предотвращения падения сервера

8. В чем разница между query parameters и path parameters?
Path parameters:
Используются для идентификации конкретного ресурса
Являются частью пути URL: /api/books/1
Обязательные для доступа к ресурсу
Обычно используются для ID или уникальных идентификаторов

Query parameters:
Используются для фильтрации, сортировки, пагинации
Добавляются после знака ? в URL: /api/books?page=1&limit=10
Необязательные параметры
Могут иметь multiple values: ?genre=fiction&genre=sci-fi
